# 将项目向量数据库迁移到 Milvus 的实施计划

## 目标
将当前基于 Chroma 的本地持久化向量库替换为 Milvus，并保持现有 RAG 检索能力、元数据过滤能力以及应用接口行为基本稳定。

## 当前现状

### 当前向量库实现
- 当前项目使用 `langchain-chroma` + `chromadb`。
- 向量库入口集中在 `utils/vector_store.py:1-19`：
  - `get_vector_store()` 返回 `Chroma`
  - `add_documents_to_vector_store()` 调用 `add_documents()` 写入向量
- 检索逻辑集中在 `utils/retriever.py:6-118`：
  - 相似度检索：`as_retriever(search_type="similarity")`
  - 带分数检索：`similarity_search_with_score()`
  - MMR 检索：`as_retriever(search_type="mmr")`
- 配置在 `conf/config.py:39-41` 和 `.env:16-18`：
  - `EMBEDDING_MODEL_NAME`
  - `CHROMA_COLLECTION_NAME`
  - `CHROMA_PERSIST_DIR`
- 当前部署中没有独立向量库服务，只有 `postgres` 和 `app`，见 `docker-compose.yml:1-43`。

### 当前依赖的关键行为
项目当前不仅依赖“向量检索”，还依赖以下语义：
1. **单 collection 多租户复用**：当前所有用户和知识库共用一个 collection，通过 metadata 过滤区分。
2. **metadata 过滤**：`utils/retriever.py:6-16` 构造了 `user_id`、`knowledge_base_id` 过滤条件。
3. **稳定 metadata 字段**：
   - 文件加载阶段写入：`user_id`、`knowledge_base_id`、`document_id`、`file_name`、`file_type`、`source`
   - chunk 切分阶段写入：`chunk_id`、`chunk_index`、`page_no`、`start_offset`
4. **固定 embedding 模型**：`sentence-transformers/all-mpnet-base-v2`，见 `conf/config.py:39`。
5. **显式 chunk_id 作为向量 ID**：见 `utils/vector_store.py:18-19`。

## 迁移原则
1. **先兼容、后优化**：第一阶段优先做到行为等价，不同时重构检索架构。
2. **保留现有 metadata 结构**：避免影响 `rag_service`、`entry_extractor` 等上游逻辑。
3. **先跑通 Milvus，再考虑分区/多 collection 优化**。
4. **明确重建索引方案**：Chroma 持久化文件不能直接迁移到 Milvus，需要重建向量索引。

## 推荐方案

### 方案 A：继续使用 LangChain 抽象，对接 Milvus（推荐）
使用 `langchain-milvus` 或其等价 LangChain 集成层，尽量保持 `get_vector_store()` / `retriever` 的调用方式不变。

优点：
- 改动集中在 `utils/vector_store.py`、`utils/retriever.py`
- 对现有 `rag_service`、`index_service` 影响较小
- 更适合当前项目已有的 LangChain 调用方式

风险：
- 需要验证 Milvus 集成对以下能力的支持是否与当前一致：
  - metadata filter
  - `similarity_search_with_score`
  - `MMR`
- 分数语义可能与 Chroma 不一致

### 方案 B：直接使用 `pymilvus` 自建封装
不依赖 LangChain vector store 适配器，直接管理 collection、schema、index 和 query。

优点：
- 可控性更高
- 后续做多租户、分区、复杂过滤会更灵活

缺点：
- 改造面更大
- 需要自己补齐 LangChain retriever 兼容层
- 超出当前“替换存储后端”的最小改动目标

**结论**：第一阶段采用方案 A，后续如需要更强控制力再切到方案 B。

## Milvus 目标设计

### 1. 连接配置
建议新增以下配置项：
- `VECTOR_BACKEND=milvus`
- `MILVUS_URI`
- `MILVUS_TOKEN`（如有认证）
- `MILVUS_COLLECTION_NAME=document_chunks`
- `MILVUS_DIMENSION`（与 embedding 维度一致）
- `MILVUS_INDEX_TYPE`
- `MILVUS_METRIC_TYPE`
- `MILVUS_SEARCH_PARAMS`

如果希望兼容未来多后端切换，也可以保留统一命名：
- `VECTOR_COLLECTION_NAME`
- `VECTOR_STORE_URI`

但从当前项目规模看，先用 Milvus 专用配置更直接。

### 2. Collection schema
Milvus collection 至少应包含：
- 主键：`chunk_id`（字符串或显式主键字段）
- 向量字段：embedding
- 标量字段：
  - `user_id`
  - `knowledge_base_id`
  - `document_id`
  - `file_name`
  - `file_type`
  - `source`
  - `chunk_index`
  - `page_no`
  - `start_offset`

重点是 `user_id` 和 `knowledge_base_id` 必须支持过滤查询。

### 3. 索引与检索策略
第一阶段建议：
- 保持当前单 collection 模型
- 保持 top-k 检索接口不变
- 保留 metadata filter
- 先使用最稳定的 similarity 检索
- MMR 如果适配层不稳定，可作为第二优先级验证项

### 4. 多租户策略
当前代码语义是：
- 所有 chunk 写入同一个 collection
- 依靠 metadata 过滤进行隔离

迁移时先保持这个模型，避免一次性引入：
- 每用户 collection
- 每知识库 collection
- partition 粒度重构

后续如果数据规模增长，再评估：
- 按 `knowledge_base_id` 建 partition
- 或按租户拆 collection

## 数据迁移策略

### 核心结论
Chroma 本地持久化目录 `storage/vector_store` 不能直接迁移到 Milvus，必须重建索引。

### 推荐迁移路径

#### 路径 1：从原始文档重新索引（推荐）
流程：
1. 保留 PostgreSQL 中 document/chunk 数据
2. 读取原始文件
3. 按现有 splitter 重新切块
4. 重新生成 embedding
5. 写入 Milvus

优点：
- 最符合当前业务逻辑
- metadata 最完整
- 不依赖 Chroma 内部存储格式

缺点：
- 耗时较长
- embedding 成本更高

#### 路径 2：基于已存 chunk 记录重建
如果 PostgreSQL 中 chunk 表已保存完整文本和 metadata，可基于 chunk 表直接重建向量，避免重新解析原始文件。

优点：
- 速度更快
- 不依赖文件加载/解析稳定性

前提：
- 需要确认 chunk 表里是否有完整文本内容和必要 metadata

**建议**：优先评估“从 chunk 表重建”是否可行；若字段不完整，再退回“从原始文件重建”。

## 实施步骤

### 阶段 1：确认技术选型与部署方式
需要先定两个关键决策：
1. **Milvus 部署方式**
   - 本地/测试：Docker Compose 自托管
   - 生产：托管版或独立部署
2. **接入方式**
   - LangChain-Milvus 适配层
   - 直接 `pymilvus`

本计划默认：**Docker Compose 自托管 + LangChain-Milvus**。

### 阶段 2：引入配置与依赖
1. 修改 `requirements.txt`
   - 增加 Milvus 相关包
   - 移除 `langchain-chroma`、`chromadb`
2. 修改 `conf/config.py`
   - 增加 Milvus 配置项
   - 去掉 Chroma 专用字段
3. 修改 `.env`
   - 补充 Milvus 连接配置

### 阶段 3：替换向量库初始化层
1. 重写 `utils/vector_store.py`
   - `get_vector_store()` 改为返回 Milvus store
   - 写入 `ids` 时继续使用 `chunk_id`
2. 确保 collection/schema/index 能自动初始化或显式初始化
3. 确保 embedding function 与当前模型一致

### 阶段 4：适配检索层
1. 修改 `utils/retriever.py`
   - 将 metadata 过滤转换为 Milvus 支持的查询表达式或过滤参数
2. 验证以下三个接口：
   - `get_retriever()`
   - `retrieve_documents_with_scores()`
   - `get_mmr_retriever()`
3. 如果 MMR 不可用：
   - 暂时降级为 similarity
   - 在代码中显式标注兼容差异

### 阶段 5：补充迁移/重建索引能力
新增一个可重复执行的重建流程，建议做成脚本或管理命令：
1. 清空目标 Milvus collection（仅针对迁移环境）
2. 遍历已有 document/chunk 数据
3. 重新生成向量并写入 Milvus
4. 输出成功/失败统计

建议支持：
- 全量重建
- 按 `knowledge_base_id` 局部重建
- dry-run 模式

### 阶段 6：更新部署编排
如果使用自托管 Milvus：
1. 在 `docker-compose.yml` 中新增 Milvus 服务
2. 配置 app 对 Milvus 的连接
3. 增加健康检查与启动依赖
4. 明确数据卷挂载策略

### 阶段 7：回归测试
至少验证以下场景：
1. 文档上传 -> 切块 -> 向量写入成功
2. 指定 `user_id` 检索结果不串租户
3. 指定 `knowledge_base_id` 检索结果正确
4. `retrieve_documents_with_scores()` 返回结构可用
5. `rag_service` 生成答案时 source 引用不丢失
6. 老数据重建后可正常检索
7. 容器环境下应用能连通 Milvus

## 风险与应对

### 风险 1：metadata filter 语法不兼容
应对：
- 在 `retriever` 层增加后端适配
- 使用集成测试固定 `user_id` / `knowledge_base_id` 过滤行为

### 风险 2：score 含义变化
应对：
- 明确 Milvus 返回的是 distance 还是 similarity
- 调整调试脚本和任何阈值逻辑

### 风险 3：MMR 不兼容或行为变化
应对：
- 首先验证
- 若不稳定，短期降级为普通相似度检索

### 风险 4：SQL 与向量库写入不一致
当前流程本身就存在该问题，迁移时建议顺手增强：
- 向量写入失败时不要把 document 标记为 `indexed`
- 为重试与补偿逻辑预留入口

### 风险 5：自托管 Milvus 部署复杂度高
应对：
- 开发环境先用最小可运行方式
- 生产再根据资源与可维护性决定是否使用托管服务

## 建议的执行顺序
1. 确认部署方式：自托管 Milvus 还是托管 Milvus
2. 确认接入层：LangChain-Milvus 还是 `pymilvus`
3. 改配置和依赖
4. 改 `vector_store.py`
5. 改 `retriever.py`
6. 跑最小联调
7. 编写重建索引脚本
8. 跑回归测试
9. 更新 README / 部署文档
10. 切换正式环境

## 预计改动范围

### 低风险改动
- 配置项替换
- 依赖替换
- README 更新

### 中风险改动
- `vector_store.py` 替换
- `retriever.py` filter 适配
- MMR / score 行为校准

### 高风险改动
- 历史数据重建
- Docker Compose 自托管 Milvus 接入
- 生产环境切换与回滚策略

## 建议的验收标准
迁移完成后满足以下条件即视为达标：
1. 应用不再依赖 Chroma
2. 新上传文档可写入 Milvus
3. 按 `user_id` / `knowledge_base_id` 的检索结果正确
4. `rag_service` 产出内容和来源字段结构不变
5. 历史知识库数据可通过重建流程恢复检索能力
6. 本地与容器环境都能正常运行

## 决策点
当前最需要先拍板的是：
1. **Milvus 采用自托管还是托管服务**
2. **第一阶段是否接受“先只保证 similarity 检索，MMR 后补”**
3. **历史数据准备从原始文件重建，还是优先从 chunk 表重建**
