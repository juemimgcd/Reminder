# 基于 FastAPI 的 Agent 项目开发规范

副标题：面向 AI 生成 `target.md` 的轻量蓝图文档

## 1. 文档用途

这份文档不是 README，也不是某个具体项目的实现说明。

它的用途只有一个：

让 AI 在接到一个新的 Agent 项目需求后，先理解应该怎么拆项目，再生成一个 `target.md`。

`target.md` 必须是项目计划，而不是功能散文。它至少要包含：

1. 项目定位
2. 核心链路
3. 分层设计
4. 目录结构
5. 阶段计划
6. 可选增强建议

## 2. 这份规范要保留什么，不保留什么

从当前项目里，真正值得保留的是“拆项目的方式”，不是具体业务名，也不是某一套固定模块清单。

要保留的是：

1. 按职责边界分层。
2. 路由层保持薄。
3. 把 Agent 能力拆成可复用节点。
4. 把结构化生成能力拆成稳定流水线。
5. 目录结构由需求倒推，而不是先把模块铺满。

不要默认保留的是：

1. 数据库
2. 用户登录
3. 文件上传
4. 知识库
5. 向量库
6. `profile`
7. `growth`
8. `advice`
9. `companion`

这些都只能按需求决定是否生成。

## 3. AI 在生成 `target.md` 前必须先判断的事

AI 在输出目录树之前，必须先判断下面这些问题：

1. 这个项目的核心目标是什么。
2. 它是问答型 Agent、工作流型 Agent、分析型 Agent，还是混合型。
3. 它是否需要 API 服务化。
4. 它是否需要结构化输出。
5. 它是否需要外部上下文输入。
6. 它是否需要状态持久化。
7. 它是否需要鉴权和多用户边界。
8. 它是否需要文件接入、检索和索引。

只有这些问题先想清楚，`target.md` 才不会默认长成一个“数据库 + 登录 + RAG 全家桶”。

## 4. 核心分层思维

这里讲的是“常见职责边界”，不是强制所有目录都必须存在。

### 4.1 `router` 层

`router` 层是入口层，负责：

- 接收请求
- 参数校验
- 调用下游模块
- 返回统一结果

这一层不负责：

- 写 Prompt
- 直接调底层模型 SDK
- 堆大段业务编排

如果项目没有 HTTP API，这一层可以不存在。

### 4.2 `utils` 层

`utils` 是 Agent 项目的核心能力层。

这里不要理解成“杂项工具”，而要理解成“能力节点目录”。

这一层最值得保留的思维是：

- 把复杂能力拆成若干个可复用节点
- 每个节点只负责一类事情
- 节点之间按流水线组合

在 Agent 项目里，`utils` 往往比 `crud` 更核心。

### 4.3 `schemas` 层

如果项目存在明确的输入输出结构，建议保留 `schemas`。

这一层负责：

- 请求模型
- 响应模型
- 结构化输出模型

如果项目只是内部脚本式 Agent，也可以不单独拆 `schemas`，但只要涉及 API、结构化结果或多模块协作，最好保留。

### 4.4 `conf` 层

`conf` 用来收口配置，而不是承载业务逻辑。

这一层通常包括：

- 环境变量
- 日志配置
- 第三方服务配置

如果项目非常轻量，也可以只有一个 `config.py`。

### 4.5 `crud` 层

`crud` 不是必选层。

只有在项目确实存在持久化数据访问时，才建议生成 `crud`。

如果项目只是：

- 单轮 Agent
- 工作流编排
- 调用外部工具
- 临时内存态处理

那么完全可以没有 `crud/`。

## 5. `utils` 层的核心抽象

这份规范最重要的部分，不是数据库，也不是用户系统，而是 `utils` 的拆法。

从当前项目里最应该抽出的通用模式是：

`llm + prompt + builder`

这不是某个具体业务模块，而是一种稳定的 Agent 能力拆分方式。

应该这样理解：

- `llm.py`
  - 统一提供模型实例和模型配置
- `<domain>_prompt.py`
  - 只定义这个领域需要的 Prompt
- `<domain>_builder.py`
  - 负责整理输入、绑定 parser、调用 prompt 和 llm、输出稳定结果

所以像当前项目里的：

- `profile_prompt.py`
- `profile_builder.py`

真正应该抽象成：

- `<domain>_prompt.py`
- `<domain>_builder.py`

如果新项目要做的是任务规划、日报生成、故障分析、会议总结，那么只需要把领域名替换掉，不要把 `profile` 原样搬过去。

## 6. Agent 项目里最稳定的三种流水线

### 6.1 结构化生成型

适用于：

- 画像
- 报告
- 建议
- 计划
- 总结
- 审核结论

推荐抽象为：

`input -> prompt -> llm -> parser -> normalized result`

推荐文件：

- `<domain>_prompt.py`
- `<domain>_builder.py`
- `schemas/<domain>.py`

建议保留的风格：

- 输入先整理成稳定字典或 JSON 文本
- 输出走结构化解析
- Builder 返回业务可直接消费的结果，而不是裸模型响应

### 6.2 检索问答型

适用于：

- 基于上下文回答问题
- 资料召回后回答
- 知识片段整合回答

推荐抽象为：

`retrieve -> format_context -> prompt -> llm -> answer`

如果项目根本不做检索，这条链路就不要出现。

### 6.3 上下文整理型

适用于：

- 原始记录整理
- 多来源信息归并
- 时间线组织
- 主题聚合
- LLM 前的数据清洗

推荐抽象为：

`raw data -> organizer -> builder input -> prompt/llm -> result`

这条链路在很多 Agent 项目里都比数据库更稳定。

## 7. 目录结构应该怎么生成

`target.md` 里的目录结构，应该先给“最小骨架”，再给“可选增强”。

不要一上来默认生成完整大树。

推荐最小骨架写法：

```text
.
├─ routers/                  # 可选：如果项目对外提供 HTTP API
├─ schemas/                  # 可选：如果项目有明确输入输出模型
├─ utils/
│  ├─ llm.py                 # 可选：如果项目依赖大模型
│  ├─ <domain>_prompt.py     # 可选：结构化生成能力
│  ├─ <domain>_builder.py    # 可选：结构化生成能力
│  ├─ <domain>_organizer.py  # 可选：上游上下文整理
│  └─ <domain>_service.py    # 可选：跨节点编排
├─ conf/                     # 可选：配置收口
├─ docs/                     # 可选：流程文档、设计说明
├─ main.py                   # 可选：FastAPI 入口
└─ target.md
```

然后在 `target.md` 里明确说明：

1. 哪些目录是当前项目必需的。
2. 哪些目录是按需添加的。
3. 为什么当前项目不需要某些常见模块。

## 8. 代码风格应抽象什么

从当前项目里，真正值得抽象的编码习惯有这些：

### 8.1 函数化优先

优先用小函数组织能力，而不是一开始就做很重的类层级。

常见命名风格：

- `get_xxx`
- `list_xxx`
- `build_xxx`
- `extract_xxx`
- `generate_xxx`
- `run_xxx`

### 8.2 路由只做薄编排

如果有 `router`，它更像入口协调器，不是业务黑盒。

### 8.3 结构化输入输出

只要涉及 Agent 结果复用，尽量把输入输出做成稳定结构，而不是散乱字符串。

### 8.4 上下文先整理，再交给 Builder

这一点比“是否上数据库”更值得保留。

抽象方式是：

`raw context -> organizer -> builder`

### 8.5 模块按职责命名，不按想象命名

例如：

- `prompt`
- `builder`
- `organizer`
- `service`

这些命名表达的是职责，迁移性更强。

## 9. `target.md` 的最低输出要求

AI 基于本规范生成 `target.md` 时，至少要包含下面这些部分：

~~~~md
# <项目名>

## 1. 项目定位
- 一句话目标
- 核心用户
- 非目标

## 2. 核心能力
- 当前项目必选能力
- 可选增强能力

## 3. 核心链路
- 用文字描述主链路
- 必要时补 1 个 Mermaid 图

## 4. 分层设计
- 当前项目需要哪些层
- 每层分别负责什么
- 哪些层本项目不需要

## 5. 目录结构
~~~text
<当前项目实际需要的目录树>
~~~

## 6. 关键模块说明
- 每个重点模块负责什么
- 为什么这样拆

## 7. 开发阶段计划
- Phase 1
- Phase 2
- Phase 3

## 8. 可选增强建议
- 如果后续需要持久化，可以怎么加
- 如果后续需要鉴权，可以怎么加
- 如果后续需要文件接入和检索，可以怎么加
~~~~

注意：

- `target.md` 必须先写当前项目真正需要的内容。
- 可选模块统一放在最后的“增强建议”里。
- 不要把建议模块写成默认模块。

## 10. 按需增强建议

这一节只作为建议，不作为默认生成项。

### 10.1 如果项目需要持久化，再加 `crud + database`

适用场景：

- 需要保存用户数据
- 需要保存任务记录
- 需要保存对话、文档、结果物

这时再考虑增加：

- `conf/database.py`
- `crud/`
- `models/`

如果项目只是一次性执行或短流程任务，不要默认加这一套。

### 10.2 如果项目需要多用户边界，再加登录和鉴权

适用场景：

- SaaS API
- 平台型系统
- 有用户隔离要求

这时再考虑增加：

- `routers/auth.py`
- `schemas/auth.py`
- `utils/auth.py`
- `utils/security.py`

如果项目只是个人 Agent、内部工具或单用户系统，不要默认加登录。

### 10.3 如果项目需要文件接入，再加 `file_loader`

适用场景：

- 需要读取 PDF、TXT、Markdown、Word 等文件
- 需要上传资料并解析

这时再考虑增加：

- `utils/file_loader.py`

如果项目的输入本来就是接口参数、消息文本或外部 API 返回值，不要默认加 `file_loader.py`。

### 10.4 如果项目需要检索式 RAG，再加检索和索引模块

适用场景：

- 需要知识库问答
- 需要向量检索
- 需要文档切分和索引

这时再考虑增加：

- `utils/text_splitter.py`
- `utils/retriever.py`
- `utils/rag_service.py`
- `utils/index_service.py`
- `utils/embeddings.py`
- `utils/vector_store.py`

如果项目不是知识库型 Agent，不要默认生成这一组。

### 10.5 如果项目需要领域结果页，再加多个 `<domain>_prompt + <domain>_builder`

适用场景：

- 一个项目里有多个结构化能力
- 例如分析、建议、计划、总结分别独立

这时再考虑增加：

- `analysis_prompt.py + analysis_builder.py`
- `plan_prompt.py + plan_builder.py`
- `summary_prompt.py + summary_builder.py`

不要把所有能力塞进一个大而混乱的 `agent_service.py`。

## 11. 一句话总结

生成 FastAPI Agent 项目计划时，优先抽象稳定的职责边界，尤其突出 `utils` 中的 `llm + prompt + builder` 和 `raw context -> organizer -> builder` 这两类通用模式；数据库、登录、文件接入、检索索引都只作为按需增强建议放在末尾，而不是默认前置。
