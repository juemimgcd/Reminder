# Mneme 前端 Obsidian 风格审美升级设计

## 1. 目标

将现有 Mneme 前端升级为简洁、精致、优雅的知识工作台。产品气质参考 Obsidian 的专注型工具体验，但不复制其具体界面；设计重点是清晰的信息层级、克制的视觉语言、连续的工作流和可靠的跨设备体验。

本次改造覆盖整个 `app/mneme_frontend_v0.2.1` 前端，包括：

- 桌面、平板和手机端的响应式工作台布局；
- 浅色、深色与跟随系统主题；
- 简体中文和英文切换；
- 研究空间、知识库、知识图谱、AI 实验室和设置等现有视图；
- 加载、空数据、错误、禁用和执行中状态；
- 组件拆分与统一设计基础组件。

现有后端接口、业务能力和预览模式保持兼容。本次不新增与审美升级无关的业务功能。

## 2. 现状与问题

当前前端已经具备完整的业务工作台，但存在以下影响体验的问题：

- 同时混用了 Obsidian、Stitch、玻璃拟态和发光卡片等视觉语汇；
- 黑紫配色面积偏大，强调色没有形成明确的使用规则；
- 页面层级主要依赖卡片、渐变和阴影，内容本身的优先级不够清晰；
- `App.vue` 承载了大部分界面，体积约 7 万字符，组件边界不清晰；
- 手机端沿用桌面多栏宽度，出现横向裁切，无法形成可用的移动工作流；
- 文案散落在模板中，缺少统一国际化层；
- 主题、布局和业务状态耦合，后续维护成本较高。

## 3. 设计原则

### 3.1 内容优先

界面首先服务于文件、知识关系、对话和研究内容。品牌区域、装饰背景和非必要卡片应收敛，主工作区获得最大可用空间。

### 3.2 克制的层级

区域层级主要使用背景明度、间距、排版和 1px 边框表达。阴影仅用于菜单、浮层、对话框和移动抽屉。避免随机渐变、持续发光和重复悬浮卡片。

### 3.3 工具感而非仪表盘感

研究空间、文件树、标签页、图谱画布和 AI 会话应形成连续工作流，不将每项信息包装成独立数据卡片。

### 3.4 跨设备重组

响应式设计不是缩小桌面界面，而是根据设备重新组织导航、面板和操作优先级。任何宽度下都不得依赖横向滚动访问主要功能。

### 3.5 可访问与可预测

所有交互元素具备 hover、focus-visible、active 和 disabled 状态。颜色不是唯一的状态表达方式。动画短促、低调，并尊重 `prefers-reduced-motion`。

## 4. 信息架构与响应式布局

### 4.1 桌面端

桌面工作台由四个可组合区域构成：

1. 活动栏：固定窄栏，提供研究空间、知识库、图谱、AI 和设置入口；
2. 资源栏：展示当前模块的文件树、会话列表或筛选器，可折叠和调整宽度；
3. 主工作区：承载文档、图谱、对话或设置内容，支持工作区标签；
4. 上下文面板：按需展示文档属性、引用来源、节点详情或模型信息，可折叠。

底部状态栏展示同步、索引、连接状态和当前空间等低优先级信息。

### 4.2 平板端

保留活动栏和主工作区。资源栏与上下文面板改为覆盖式抽屉，不永久挤压主内容。抽屉支持键盘关闭、点击遮罩关闭和焦点约束。

### 4.3 手机端

主模块入口改为底部导航。资源浏览、会话列表、设置分组等次级内容使用全屏抽屉或独立页面。页面顶部仅保留返回、标题和当前上下文操作。图谱和编辑器占满可用宽度，不出现桌面侧栏残留或横向裁切。

## 5. 视觉系统

### 5.1 色彩

主题使用语义化令牌，不允许组件直接依赖具体灰阶编号。

- 深色主题：冷调炭黑背景、石墨灰表面、柔和浅灰正文；
- 浅色主题：柔和纸白背景、暖灰表面、低刺激深灰正文；
- 强调色：克制的紫色，用于主要操作、当前选中项、焦点环和关键图谱节点；
- 状态色：成功、警告、错误和信息颜色分别定义浅色与深色版本；
- 边框与分隔线：低对比但可辨识，避免依赖阴影划分常驻区域。

主题需要分别校准文字、边框、选中态、输入框、浮层和图谱元素，不能只做颜色反转。

### 5.2 排版

- 界面与正文使用清晰的无衬线字体，并保留中文系统字体回退；
- 长文标题可使用更具阅读感的字体层级，但不引入装饰性展示字体；
- 代码、日期、快捷键和技术元数据使用等宽字体；
- 桌面端保持紧凑密度，移动端提高行高和触控区域；
- 标题、正文、辅助信息和元数据形成稳定的字号与字重阶梯。

### 5.3 形状与动效

- 常规控件使用小到中等圆角，避免大面积胶囊化；
- 常驻面板不使用重阴影；
- 面板开合、主题切换、标签切换使用 120–200ms 的缓动；
- 图标统一使用 Lucide 线性图标，规范尺寸、线宽和对齐方式。

### 5.4 品牌呈现

将当前较醒目的 “Mneme Intelligence / Cognitive Sanctuary” 收敛为紧凑产品标识。品牌不占用大块导航空间，研究内容与当前工作上下文优先。

## 6. 页面设计

### 6.1 研究空间与知识库

资源栏使用标准树形结构展示目录、文档、索引状态和快捷操作。主工作区在没有选中文档时展示有明确下一步的欢迎或空状态；选择文档后进入阅读与内容工作区。上传、创建和同步作为上下文相关的主要操作出现。

### 6.2 知识图谱

图谱保留沉浸式画布。搜索、过滤、缩放、重新布局和视图切换集中在稳定工具区。节点详情在右侧上下文面板或移动端底部抽屉中展示。节点标签应避免重叠，悬停、选中和关联路径具备清晰但克制的状态差异。

### 6.3 AI 实验室

桌面端使用会话列表、对话正文和来源/模型上下文的分栏结构。聊天历史可折叠，消息正文提升可读性，引用来源与系统状态不与正文争夺视觉焦点。移动端会话列表与对话页分离。

### 6.4 系统设置

设置页按外观、语言、AI 模型、数据同步和账户分组。主题与语言即时生效；保存型操作显示执行中、成功和失败反馈。危险操作与普通偏好设置在视觉上明确隔离。

## 7. 组件与代码边界

`App.vue` 只负责应用外壳、路由式视图选择和顶层布局组合。界面拆分为以下层级：

- `components/shell`：活动栏、资源栏、工作区标签、上下文面板、状态栏、移动导航；
- `components/ui`：按钮、输入框、选择器、菜单、对话框、提示条、骨架屏、空状态和错误面板；
- `views`：研究空间、知识库、知识图谱、AI 实验室和设置；
- `composables`：业务工作区状态、外观偏好、国际化和响应式面板状态；
- `locales`：`zh-CN` 与 `en-US` 翻译资源；
- `styles`：语义设计令牌、基础样式和少量共享布局工具。

优先复用现有 Vue、Lucide、D3 和 Tailwind 能力，不额外引入大型 UI 框架。

## 8. 主题与语言状态

新增独立偏好设置层：

- 主题模式为 `light`、`dark` 或 `system`；
- 语言为 `zh-CN` 或 `en-US`；
- 用户选择保存到 `localStorage`；
- 首次访问默认跟随系统主题，并根据浏览器语言选择中文或英文；
- `system` 模式实时响应系统主题变化；
- 切换主题或语言不会重新请求无关业务数据，也不会清空当前工作区。

所有用户可见文案通过翻译键读取。日期、数量、状态文本和错误提示使用当前语言的格式化规则。后端返回的不可翻译原始错误作为详细信息保留，界面提供本地化摘要。

## 9. 数据状态与错误处理

远端数据视图统一覆盖以下状态：

- 首次加载：使用与最终布局一致的骨架屏；
- 空数据：解释当前状态并提供明确主要操作；
- 局部刷新：保留已有内容并显示非阻塞进度；
- 请求失败：展示本地化摘要、重试操作和必要的技术详情；
- 执行中：相关按钮禁用并显示进度，防止重复提交；
- 危险操作：需要确认，并清楚描述影响范围。

现有 `useMnemeWorkspace` 保留业务数据职责，但主题、语言和纯布局状态迁移到独立 composable，避免继续扩张单一状态容器。

## 10. 验收标准

### 10.1 功能与构建

- `npm run lint` 通过；
- `npm run build` 通过；
- 现有 Playwright 测试通过；
- 新增主题切换与持久化、语言切换、响应式导航测试；
- 后端 pytest 回归测试通过。

### 10.2 视觉与响应式

- 在桌面、平板和手机宽度分别检查浅色与深色主题；
- 手机端不出现主要内容横向裁切或桌面侧栏残留；
- 主要页面均具备加载、空数据和错误状态；
- 交互元素具备 hover、focus-visible、active 和 disabled 状态；
- 浅色和深色主题均保持清晰对比度；
- 中文和英文下不出现关键按钮截断、布局跳动或不可访问操作；
- 动画尊重 `prefers-reduced-motion`。

### 10.3 范围约束

- 不修改现有后端 API 契约；
- 不删除当前业务能力；
- 不为视觉升级引入新的大型组件库；
- 不把页面重做成卡片密集的通用 SaaS 仪表盘；
- 不复制 Obsidian 的商标、品牌资产或专有界面细节。

## 11. 交付方式

实现分阶段进行：先建立主题、国际化与应用外壳，再迁移各业务视图，最后完善状态、响应式和浏览器视觉验证。每个阶段保持可构建、可测试，避免一次性替换后难以定位回归。
